\Chapter{Tervezés}

A tervezési fejezet célja az Acxor rendszer architektúrájának, adatmodelljének és felhasználói felületének részletes bemutatása. Ebben a fejezetben kerülnek ismertetésre azok a döntések, amelyek a fejlesztés irányvonalát meghatározták, beleértve a technológiai stack kiválasztását, az adatbázis-séma kialakítását, az API tervezését és a komponensstruktúrát. A tervezési fázis különösen fontos volt, hiszen a rendszernek három platformon – webes böngészőben, mobil eszközön és szerver oldalon – egyidejűleg kellett megfelelően működnie, miközben közös üzleti logikát és adatréteget használt.

\Section{Rendszerarchitektúra}
% ─────────────────────────────────────────────────────────────────────────────

Az Acxor rendszer három jól elkülönülő részből áll: a webes frontend alkalmazásból, a React Native alapú mobil alkalmazásból és a Node.js backend API-ból. Mindhárom rész egyetlen közös RESTful API-on keresztül kommunikál, amelyet a backend szolgáltat. Ez a megközelítés lehetővé teszi, hogy a kliens oldalak teljesen független életet éljenek, miközben ugyanazokat az adatokat, autentikációs mechanizmust és üzleti szabályokat alkalmazzák. Az architektúra tehát klasszikus kliens-szerver modellt követ, azzal a kiegészítéssel, hogy két teljesen különböző típusú kliens is ugyanahhoz a backendhez csatlakozik. A rendszer magas szintű architektúrája a \ref{fig:sys-architecture}. ábrán látható.

\centering
\includegraphics[width=0.8\textwidth]{images/acxor_architecture_v2.png}

A backend alkalmazás Express.js keretrendszeren alapul, és a következő főbb feladatokat látja el: felhasználói hitelesítés (JWT), projektek, feladatok és alfeladatok kezelése, sprintek és napló bejegyzések tárolása, XP-alapú gamifikációs logika végrehajtása, valamint GitHub API-val való integráció. Az adatok MongoDB adatbázisban tárolódnak, amelyhez a Mongoose ODM biztosít séma-alapú hozzáférést.

A webes frontend Vite segítségével épített React alkalmazás, amely Tailwind CSS-t használ a stílusok kialakítására. Az állapotkezelés React Context API-n és \texttt{useReducer} hookson alapul, amelyek Redux-szerű, de külső könyvtártól független megoldást nyújtanak. A mobil alkalmazás Expo keretrendszerre épített React Native projekt, amely ugyanolyan üzleti logikát követ, mint a webes változat, azonban natív mobil UI komponenseket alkalmaz.

\Section{Technológiai stack}
% ─────────────────────────────────────────────────────────────────────────────

A fejlesztés során a technológiai döntések több szempontot is figyelembe vettek: a fejlesztői ökoszisztéma érettségét, a közösségi támogatottságot, a típusbiztonságot, valamint azt, hogy a választott technológiák lehetővé tegyék a gyors prototípuskészítést anélkül, hogy a kód minőségét veszélyeztetnék.

\SubSection{Backend technológiai döntések}

A backend Node.js és TypeScript kombinációjára épül \cite{typescript}. A TypeScript alkalmazása biztosítja a statikus típusellenőrzést, ami különösen hasznos az API végpontok bemeneti és kimeneti adatainak validálásakor. Az Express.js a de facto standard REST API keretrendszer Node.js alatt, amelynek middleware-alapú felépítése rugalmasan bővíthető. A MongoDB dokumentumorientált adatbázis jól illeszkedik a projektmenedzsment adatok természetéhez, ahol a rugalmas sémák lehetővé teszik a feladatok, alfeladatok és projektek összetett struktúráinak tárolását. A Mongoose ODM séma validációt és referencia-alapú kapcsolatokat biztosít a dokumentumok között.

A hitelesítés JSON Web Token alapon működik. Bejelentkezéskor a szerver egy 7 napos érvényességű tokent állít ki, amelyet a kliens minden API kéréshez csatol az \texttt{Authorization} fejlécben. A jelszavak bcryptjs segítségével kerülnek hashelésre, 10-es salt rounds értékkel. A védett végpontokat egy \texttt{protect} middleware védi, amely ellenőrzi a token érvényességét és azonosítja a felhasználót.

\SubSection{Webes frontend}

A webes alkalmazás React 18-ra épül \cite{react}, Vite fejlesztői szerverrel és build eszközzel \cite{vite}. A Tailwind CSS \cite{tailwind} utility-first megközelítése lehetővé tette a gyors és következetes stílusok kialakítását anélkül, hogy külön CSS fájlokat kellett volna karbantartani. Az állapotkezelés teljesen egyedi megoldást alkalmaz: a \texttt{GlobalContext} tárolja az alkalmazás fő adatait (felhasználó, projektek, feladatok, alfeladatok, sprintek), míg a \texttt{ViewContext} a nézetek közötti navigációért felelős. Az axios könyvtár \cite{axios} HTTP kérések küldésére szolgál a webes kliensben.

\SubSection{Mobil frontend}

A mobil alkalmazás React Native alapú \cite{reactnative}, Expo keretrendszerrel \cite{expo}. Az Expo lehetővé teszi a gyors fejlesztést és az eszközön való tesztelést Expo Go segítségével. A navigáció Expo Router révén \cite{exporouter} fájlrendszer-alapú, ami közelebb áll a Next.js szemléletéhez. A mobil alkalmazás a fetch API-t alkalmazza HTTP kérésekhez, és a backend ugyanazon végpontjait éri el, mint a webes kliens. Az állapotkezelés megegyezik a webes verzióéval: Context API és useReducer kombináció.

% ─────────────────────────────────────────────────────────────────────────────
\Section{Karbantarthatósági szempontok a kliens architektúrában}
% ─────────────────────────────────────────────────────────────────────────────

A háromplatformos architektúra egyik legkritikusabb tervezési kérdése az volt,
hogy hogyan lehet a webes React alkalmazás és a React Native mobil alkalmazás
között a kódot és az üzleti logikát úgy szervezni, hogy a rendszer hosszú távon
karbantartható maradjon. Ez a szempont már a tervezési fázisban meghatározó
szerepet játszott, hiszen a két kliens párhuzamos fejlesztése és jövőbeli
módosítása komoly konzisztenciaproblémákat vethet fel, ha az architektúra nem
veszi figyelembe a karbantarthatóság elveit.

\SubSection{Közös állapotkezelési minta}

A karbantarthatóság szempontjából az egyik legfontosabb döntés az volt, hogy
mindkét kliensalkalmazás ugyanazt az állapotkezelési mintát alkalmazza: React
Context API-t és \texttt{useReducer} hookot. Ez a döntés a kódban is egyértelműen
tükröződik: a webes alkalmazásban a \texttt{GlobalContext.tsx} és a
\texttt{Reducer.tsx} fájlok, a mobilon pedig a \texttt{context/GlobalContext.tsx}
és a \texttt{context/Reducer.ts} fájlok strukturálisan és névhasználatban szinte
teljesen megegyeznek. Az action típusok — \texttt{SET\_USER}, \texttt{SET\_TASKS},
\texttt{APPEND\_SUBTASKS}, \texttt{DELETE\_SPRINT} és társaik — azonos nevekkel és
azonos payload-struktúrával szerepelnek mindkét platformon. Hasonlóképpen, a
\texttt{ViewReducer} és a \texttt{ViewContext} a weben az állapotalapú navigáció
vezérlőjeként azt a mintát követi, amelynek mobilon az Expo Router
fájlrendszer-alapú navigációja felel meg: mindkét esetben egyetlen, jól
elkülönített réteg felelős a nézetváltások kezeléséért.

Ennek a döntésnek két közvetlen karbantarthatósági előnye van. Egyrészt, ha egy
fejlesztő a webes kliensben elsajátítja az állapotkezelési mintát, azonnal képes
a mobil kliensben is eligazodni és módosításokat végezni — nincs szükség egy
teljesen eltérő paradigma megismerésére. Másrészt, ha az üzleti logika változik
(például új mezők kerülnek a feladatmodellbe), az action típusok és a reducer
ágak mindkét kliensben ugyanolyan struktúrájú módosítást igényelnek, ami
csökkenti az elcsúszás (divergencia) kockázatát.

\SubSection{Kódduplikáció és annak tudatos kezelése}

A két kliens között azonban szükségszerűen van kódduplikáció. Mindkét kliens
tartalmaz egy \texttt{Actions.ts} fájlt, amelyek lényegében ugyanazokat az
aszinkron action creator függvényeket valósítják meg. Az egyetlen érdemi
különbség a HTTP-rétegben rejlik: a webes verzió az \texttt{axios} könyvtárat
alkalmazza, míg a mobil a beépített \texttt{fetch} API-t használja. Ez a
duplikáció tudatos kompromisszum: az alternatíva egy közös, megosztott csomag
kialakítása lett volna, ami monorepo struktúrát igényelt volna (például Nx
vagy Turborepo segítségével).

A monorepo megközelítés karbantarthatósági előnye vitathatatlan: a közös logikát
egyetlen helyen kellene módosítani, és a változás automatikusan érvényre jutna
mindkét kliensben. Ugyanakkor a monorepo bevezetése jelentős konfigurációs
többletmunkával jár, és egy egyszemélyes fejlesztői projekt keretein belül az
ebből fakadó komplexitás nem arányos az előnyökkel. Ezért a jelen
implementációban a két kliens egymástól független repositoryként él, a duplikált
logika pedig manuálisan szinkronizált. Ez egy tudatosan vállalt karbantarthatósági
adósság (technical debt), amelynek felszámolása a jövőbeli fejlesztések egyik
iránya lehet.

\SubSection{A közös backend mint karbantarthatósági horgony}

A két kliens közötti konzisztencia legfőbb biztosítéka a közös backend API.
Mindkét kliens ugyanazokat a végpontokat hívja, ugyanazokat a JSON struktúrákat
kapja válaszul, és ugyanazokat a hibaüzeneteket kezeli — a backend így az
egyetlen igazságforrás (single source of truth) a rendszerben. Ez a
kódban is nyomon követhető: a webes \texttt{TasksView.tsx} és a mobil
\texttt{tasks.tsx} képernyő egyaránt a \texttt{/api/tasks/project/:projectId}
végpontot hívja a feladatok betöltéséhez, és mindkettő az \texttt{ITask}
interfésznek megfelelő JSON-t kap vissza.

Ha az adatmodell változik — például egy új mező kerül a \texttt{Task}
dokumentumba —, a backend egyetlen módosítása elegendő ahhoz, hogy mindkét
kliens az új adatot megkapja; a klienseknek csak a megjelenítési réteget kell
frissíteniük. Ez az elv különösen jól érvényesül a TypeScript-alapú fejlesztés
kontextusában: a backend Mongoose modelljeihez tartozó TypeScript interfészek
(\texttt{ITask}, \texttt{IProject} stb.) elméletileg exportálhatók és a
kliensekben importálhatók lennének egy megosztott típuscsomagból. A jelenlegi
implementációban ezek a típusok mindkét kliensben kézzel vannak definiálva —
ahogy az a \texttt{TasksView.tsx} és a \texttt{tasks.tsx} fájlok
\texttt{type Task = \{ \_id: string; title: string; status: ...\}} blokkjain is
látható —, ami azt jelenti, hogy egy adatmodell-változáskor mindhárom helyen
(backend, webes kliens, mobil kliens) szükséges a frissítés. Ez az API contract
szinkronizálási kihívás szintén a jövőbeli fejlesztések egyik prioritása.

\SubSection{Tesztelhetőség mint karbantarthatósági alap}

A karbantarthatóság és a tesztelhetőség szorosan összefügg: egy jól tesztelt
rendszerben a módosítások hatása kontrollálható, a regressziók gyorsan
felismerhetők. Az Acxor architektúrájában a karbantarthatóságot szolgáló
tesztelési döntések közé tartozik a backend service réteg — \texttt{xpService},
\texttt{logService}, \texttt{githubService} — elkülönítése a route
handlererektől. Ennek eredménye, hogy az \texttt{xpService.test.ts} és a
\texttt{logService.test.ts} tesztek közvetlenül a service függvényeket hívják,
HTTP-réteg nélkül: az \texttt{addXP} függvény tesztjei például valódi
MongoDB-kapcsolaton dolgoznak, de egyetlen Express routert sem indítanak el.
Ez egyszerre növeli a tesztek megbízhatóságát és csökkenti futási idejüket.

A frontend oldalon a Context API és \texttt{useReducer} pattern szintén a
tesztelhetőséget szolgálja. A \texttt{Reducer.test.tsx} fájl tesztjei jól
illusztrálják ezt az elvet: az \texttt{AppReducer} tiszta függvény, amelynek
teszteléséhez nincs szükség se böngészőre, se mockolt hálózati rétegre —
elegendő egy egyszerű \texttt{AppReducer(initialState, \{ type: 'ADD\_TASK',
payload: task \})} hívás. Az \texttt{Actions.test.ts} tesztekben az
\texttt{axios} mockolásával pedig az action creatorok hálózati viselkedése is
ellenőrizhető anélkül, hogy valódi szerverre lenne szükség.

\SubSection{Navigációs architektúra és karbantarthatóság}

A webes és a mobil alkalmazás navigációs megközelítése szándékosan különbözik
egymástól, és ez a különbség szintén karbantarthatósági szempontból indokolt.
A webes alkalmazásban a \texttt{ViewContext} és a \texttt{ViewReducer} kezeli a
nézetek közötti váltást: az \texttt{App.tsx} egyetlen \texttt{renderView()}
függvénnyel választja ki az aktív komponenst az \texttt{activeView} állapot
alapján, külső routing könyvtár nélkül. A mobilon ezzel szemben az Expo Router
\texttt{app/(tabs)/} mappa szerkezete definiálja a navigációt: minden tab egy
önálló fájl (\texttt{tasks.tsx}, \texttt{kanban.tsx}, \texttt{log.tsx} stb.),
és a \texttt{\_layout.tsx} regisztrálja őket az egyedi \texttt{CustomTabBar}
komponenssel.

Bár ez a különbség kódduplikációt okoz a navigációs logikában, mindkét
megközelítés a saját platformján a legtermészetesebb és legkarbantarthatóbb
megoldás: a webes SPA kontextusában az állapot-alapú nézetes navigáció egyszerűbb
és kevesebb függőséget igényel, míg a mobilon a fájlrendszer-alapú routing az
Expo és React Native ökoszisztéma bevett, jól dokumentált konvenciója. Ennek
követése csökkenti a jövőbeli fejlesztők betanulási idejét, és összhangban van
azzal a tervezési elvvel, hogy minden platform a saját idiómáját kövesse —
még akkor is, ha ez rövid távon több párhuzamos kódot jelent.

% ─────────────────────────────────────────────────────────────────────────────
\Section{A karbantarthatóság értékelése}
% ─────────────────────────────────────────────────────────────────────────────

A rendszer fejlesztése során a karbantarthatóság nem utólagos szempont volt, hanem tervezési alapelv. Martin megfogalmazásában a karbantartható kód egyik legfontosabb tulajdonsága, hogy minden egység egyetlen, jól meghatározott felelősséggel rendelkezzen \cite{martin2008clean} -- ezt az elvet az Acxor backend service rétege közvetlenül tükrözi. Ennek értékelése három dimenzióban végezhető el: a kódbázis struktúrája, a web--mobil összehangoltság és a jövőbeli bővíthetőség szempontjából.

\SubSection{Kódbázis struktúrájának karbantarthatósága}

A backend kódja jól elkülönített rétegekre épül: a route handlerek kizárólag a HTTP kérések fogadásáért és a válasz küldéséért felelősek, az üzleti logika a service rétegben él (\texttt{xpService}, \texttt{logService}, \texttt{githubService}), az adathozzáférés pedig a Mongoose modelleken keresztül valósul meg. Ez a háromrétegű szétválasztás a klasszikus rétegelt architektúra (layered architecture) elvét követi \cite{gamma1994design}, amelynek lényege, hogy az egyes rétegek csak a szomszédos réteggel kommunikálnak, és a felelősségek élesen elkülönülnek egymástól. A gyakorlatban ez azt jelenti, hogy például az XP számítási logika megváltoztatásához egyetlen fájlt (\texttt{xpService.ts}) kell módosítani, és az összes érintett végpont automatikusan az új logikát alkalmazza. Az ilyen jellegű, jól lokalizálható módosíthatóság a karbantartható szoftverek egyik legfontosabb jellemzője.

A szoftver életciklusa szempontjából különösen fontos a módosíthatóság (modifiability) és a bővíthetőség (extensibility) kérdése. Fowler szerint a kód refaktorálhatósága szorosan összefügg azzal, hogy az egyes komponensek mennyire függetlenek egymástól \cite{fowler2018refactoring}: ha egy változtatás csak egyetlen, jól körülhatárolt egységet érint, a regresszió kockázata minimális. Az Acxor service rétege ezt az elvet érvényesíti, mivel az \texttt{xpService.ts} vagy a \texttt{logService.ts} módosítása nem érinti a route handlerek kódját.

A TypeScript használata mindhárom platformon szintén közvetlen karbantarthatósági előnyt jelent \cite{typescript}: a típusrendszer fordítási időben jelzi az inkompatibilis változásokat, amelyek dinamikusan típusos kódban csak futásidőben derülnének ki. Ez különösen értékes akkor, amikor az adatmodell változik, és a változásnak mindhárom platformon át kell propagálódnia.

\SubSection{Web és mobil kliens összehangoltsága}

A webes React alkalmazás és a React Native mobil alkalmazás közötti karbantarthatóság legfőbb biztosítéka a közös állapotkezelési minta. Mindkét kliens ugyanolyan struktúrájú Context-et, Reducer-t és action creator függvényeket alkalmaz, ami azt jelenti, hogy egy fejlesztő, aki az egyik kliensben eligazodik, minimális tanulással képes a másikban is hatékonyan dolgozni. Ez az egységes minta a szoftver életciklusa szempontjából is előnyös: az onboarding ideje rövidebb, és a tudásátadás kevésbé függ az egyéni fejlesztői ismeretektől \cite{martin2008clean}. A közös backend API pedig garantálja, hogy az adatmodell változásai mindkét klienshez egyszerre jutnak el, és nem lehetséges, hogy a két kliens eltérő adatszerkezettel dolgozzon.

Ugyanakkor a dolgozat őszintén elismeri a jelenlegi architektúra korlátait is. Az \texttt{Actions.ts} fájlban lévő aszinkron action creator logika, a TypeScript interfész definíciók és a hibakezelési konvenciók mindkét kliensben kézzel vannak karbantartva, ami azt jelenti, hogy egy üzleti logika változásakor két helyen kell elvégezni az azonos módosítást. Fowler terminológiájával élve ez egy tudatosan vállalt kódduplikáció \cite{fowler2018refactoring}, amelynek felszámolása -- egy megosztott csomag (shared package) bevezetésével -- a rendszer következő fejlődési lépése lehet.

\SubSection{Összegző értékelés}

Összességében az Acxor architektúrája a karbantarthatóság szempontjából egyensúlyt tart a pragmatizmus és az elvek között. A rendszer nem törekszik a tökéletes duplikáció\-mentességre, ha az azzal járó komplexitás nem arányos az előnyökkel -- de minden fontosabb architekturális döntés mögött felismerhető a karbantarthatósági szempont: a service réteg szétválasztása \cite{gamma1994design}, a közös állapotkezelési minta, a TypeScript típusbiztonság \cite{typescript} és a tesztlefedettség együtt olyan rendszert alkotnak, amelyen egy új fejlesztő a belépéstől kezdve hatékonyan tud dolgozni, és amelyen a változtatások hatása előre látható és kontrollálható \cite{martin2008clean}.

\Section{Adatbázis-tervezés}
%─────────────────────────────────────────────────────────────────────────────

Az adatbázis tervezésekor a legfontosabb szempontok a normalizált, de praktikusan használható séma kialakítása volt. A MongoDB dokumentumorientált természete lehetővé tette a referencia-alapú kapcsolatok (ObjectId hivatkozások) és a beágyazott dokumentumok vegyítését. A rendszer összesen hét Mongoose modellt alkalmaz.

\SubSection{Adatmodellek}

A \texttt{User} modell tartalmazza a felhasználói adatokat, köztük a gamifikációhoz szükséges \texttt{xp}, \texttt{level} és \texttt{prestige} mezőket. A \texttt{Project} dokumentum tárolja az adminisztrátor, a közreműködők és a megtekintők referenciáit, ezáltal biztosítva az alapszintű hozzáférésvezérlést. A \texttt{Task} modell egy projektre és opcionálisan egy sprintre mutat, emellett tartalmaz \texttt{xpReward}, \texttt{deadline} és \texttt{assignedTo} mezőket. A \texttt{Subtask} modell szülő feladathoz kapcsolódik, és hasonló szerkezetet követ. A \texttt{Sprint} modell projekthez tartozik, és dátumtartománnyal rendelkezik. A \texttt{Log} modell eseménynaplózásra szolgál, projekt-szinten. A \texttt{Mail} modell a felhasználók közötti üzenetküldést teszi lehetővé. \SubSection{XP és szintlépési rendszer tervezése}

A gamifikációs rendszer tervezésekor fontos szempont volt, hogy a szintlépési formula megfelelően skálázódjon: az alacsonyabb szinteken legyen könnyen teljesíthető, de magasabb szinteken és presztízs értéknél egyre több munkát igényeljen. A választott formula a \ref{eq:xp-formula}. egyenletben látható:

\begin{equation}
\text{XP szükséges} = 20 + (\text{következő szint} \times 10) + (\text{prestige} \times \text{szint} \times 10)
\label{eq:xp-formula}
\end{equation}

A konstansok megválasztásának indoka a fokozatos nehezedés biztosítása. Az alapértéket ($20$) azért vezettük be, hogy az első szintlépéshez is mérhető mennyiségű munka szükséges legyen. A szint-szorzó ($\times 10$) lineáris növekedést biztosít a haladás során, a prestige-szorzó ($\times \text{szint} \times 10$) pedig exponálisan növeli a követelményt presztízs elérése után. Néhány konkrét példa az értékek szemléltetéséhez:

\begin{itemize}
  \item Prestige 0, szint 1 → következő szinthez: $20 + (2 \times 10) + 0 = 40$ XP,
  \item Prestige 0, szint 4 → következő szinthez: $20 + (5 \times 10) + 0 = 70$ XP,
  \item Prestige 1, szint 1 → következő szinthez: $20 + (2 \times 10) + (1 \times 1 \times 10) = 50$ XP,
  \item Prestige 1, szint 4 → következő szinthez: $20 + (5 \times 10) + (1 \times 4 \times 10) = 110$ XP.
\end{itemize}

Ez a formula biztosítja, hogy az első szintlépés viszonylag kevés XP-vel elérhető, ugyanakkor a prestige rendszer exponálisan növeli a szintenként szükséges XP mennyiséget. Az \texttt{addXP} service függvény a szintlépési logikát egy \texttt{while} ciklusban kezeli, hogy egyszerre több szinten is lehetséges legyen lépni, ha elég XP gyűlik össze. A jutalmak a következők: befejezett feladat esetén alapértelmezetten 5 XP, befejezett alfeladat esetén 3 XP jár a felelős felhasználónak.

\SubSection{Hitelesítési folyamat}

A hitelesítés kétlépéses folyamat: regisztrációkor a szerver bcrypt segítségével hasheli a jelszót, majd JWT tokent állít ki. Bejelentkezéskor az elküldött jelszót összehasonlítja a tárolt hashsel, és sikeres egyezés esetén új tokent bocsát ki. A token tartalmazza a felhasználó azonosítóját (\texttt{id} mező), 7 napos érvényességgel. A védett middleware a tokent dekódolja, megkeresi az adatbázisban a felhasználót, és ha megtalálja, csatolja a \texttt{req.user} objektumhoz a további feldolgozáshoz.

\SubSection{Feladat-befejezési folyamat}

A feladatok befejezése nem az általános UPDATE végponton keresztül történik, hanem egy dedikált \texttt{PUT /api/tasks/:id/complete} végponton. Ez a döntés azért született, mert a befejezéshez kapcsolódó logika (XP jóváírás, naplóbejegyzés írása) komplex mellékhatásokat von maga után, amelyeket el kell különíteni az egyszerű adatmódosítástól. A befejezési végpont ellenőrzi, hogy a feladat korábban már el volt-e végezve (\texttt{alreadyDone} jelző), és csak az első befejezéskor ír jóvá XP-t. Hasonló logika érvényes az alfeladatokra is.

\Section{Állapotkezelés tervezése}
% ─────────────────────────────────────────────────────────────────────────────

Az állapotkezelés mindkét kliensalkalmazásban React Context API-ra és useReducer hookra épül. Ez a megközelítés a Redux tároló-action-reducer mintáját követi, de anélkül, hogy külső könyvtár függőséget vonna be. A webes alkalmazásban két Context létezik párhuzamosan: a \texttt{GlobalContext} és a \texttt{ViewContext}.

A \texttt{GlobalContext} a következő állapotmezőket tartalmazza: \texttt{user} (bejelentkezett felhasználó adatai), \texttt{token} (JWT token), \texttt{projects} (projektek listája), \texttt{tasks} (aktuális projekt feladatai), \texttt{subtasks} (feladatokhoz tartozó alfeladatok), \texttt{sprints} (sprintek), \texttt{selectedProject} (kiválasztott projekt), \texttt{loading} és \texttt{error} (kérés állapot jelzők).

Az \texttt{APPEND\_SUBTASKS} action típus különösen figyelemre méltó: mivel több feladat alfeladatait töltjük be párhuzamosan, a hagyományos \texttt{SET\_SUBTASKS} felülírná a korábbi eredményeket. Az \texttt{APPEND\_SUBTASKS} ezért az adott feladathoz tartozó korábbi alfeladatokat kiszűri, majd hozzáfűzi az új eredményt, így az összes betöltött alfeladat megmarad az állapotban.

A \ref{fig:state-mgmt}. ábra a teljes adatfolyamot szemlélteti: a komponens dispatch hívást küld, a reducer új állapotot állít elő, a GlobalContext Providerén keresztül frissülnek a feliratkozott komponensek, miközben az Actions.ts fájl aszinkron módon kommunikál a backend API-val.

\Section{GitHub integráció tervezése}
% ─────────────────────────────────────────────────────────────────────────────

Az Acxor egyik kiegészítő funkciója a GitHub repozitóriumokból való projektimportálás. A tervezés során el kellett dönteni, hogy ez a funkció szerver oldalon vagy kliens oldalon menjen végbe. A szerver oldali megközelítés mellett döntöttünk, mivel a GitHub API tokent nem célszerű közvetlenül a kliensből küldeni a GitHub felé: a szerver közvetítőként működik, ami lehetővé teszi a jövőbeli token validációs logika hozzáadását is.

A folyamat a következő lépésekből áll: a felhasználó megadja GitHub felhasználónevét és személyes hozzáférési tokenét, a kliens elküldi a backend \texttt{/api/github/user/\allowbreak{}repos/\allowbreak{}names} végpontra, a szerver a GitHub API segítségével lekéri a repozitóriumok listáját, majd visszaküldi a neveket. A felhasználó kiválaszt egy repozitóriumot, amelynek issue-it a backend lekéri a \texttt{/api/github/:owner/:repo/issues} végponton keresztül. Az issues-ek feladatokká alakulnak, és egy projektlétrehozási hívással együtt kerülnek az adatbázisba.
